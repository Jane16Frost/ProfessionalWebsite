A-Maze-Game

Designing and Creating Features

A Maze Game is a simple project idea, I wanted to create to challenge my skills. The concept was making a game for a player to solve a maze. I began with drawing out assets to be able to test and ensure my game was working properly. As seen in Figure 1, I created six tiles to represent the possible directions a player could go. The six tiles can represent all possibilities by being rotated to the correct orientation. I also created a player, and rabbit icon to give the user feedback and a goal. The point of the game is to catch rabbits in the maze. As a later addition I also added directional arrows to give players a hint on what direction they needed to go in, with the arrows pointing to the rabbit.

Algorithms and Implemntation

In the process of coding a maze game, there were two core challenges. Firstly how to represent the possible paths in a given square. The second was a maze generation algorithm.

Respresenting the squares is a very important detail in the creation of mazes. There are many possible way of doing it each with their own challenges. The way that first comes to mind is to have the spuares either being open or blocked off. The problem with this method is that drawing the game and moving the player require the computer to check all remaining squares. This is why I decided to represent the players options with only one check. I acheive this by assigning each tile a number, made out of four possible prime numbers each of which having their own direction. By multiplying these numbers, I created one number and used a switch case statement to draw each tile with its correct image and orientaion. I was also able to implement character movement by checking that the tile under them had a remainder of zero when divided by the prime number corresponding to the direction the player is trying to move in.

To solve the second problem, I researched different types of maze generation algoritms. The algorithm I implemented was the Hunt and Kill method. To preform this kind of generation, the computer begins by picking a random square. It then preforms a random "walk" around the grid through squares that havent been visited until it has no unvisited neighbors. This allows the computer to create dead ends in the maze, and is the kill part of the algorithm. The Hunt part begins by searching the grid to find a square that hasnt been visited but has a visited neighbor. Once found the computer will bridge a between the squares and enter back into kill mode. Since the computer is always conecting back to the original path the player will be able to reach any square in the maze. This was an important feature, becuase I wanted the player and objective to randomly spawn in the maze. If the player couldn't reach all squares the possibility of an unsolvable level arises.

These core problems were very fun to solve, and make this a project Im very proud of. The rest of the code was easy to implement and made for an interesting game experience. I hope to finish this project one day, including the story mode and be able to properly show it off.
